本文档定义如何确保在撰写过程中严格遵循既定的大纲、设定和规则，防止写飞/写废。

## 一、核心保障机制

### 1.1 三大铁律

| 铁律 | 内容 | 违反后果 |
|------|------|----------|
| **严格按方案** | 写手必须严格按照 outline.md、characters.md、world.md、skills.md 等设定写作 | 审核打回 |
| **审核时暂停** | 写完5章后必须暂停，等编辑审核完成后才能继续下一批 | 暂停写作 |
| **双重确认** | 编辑审核通过后，总编必须做第二重确认 | 不得进入下一批 |

### 1.2 质量门体系

| 检查点 | 检查内容 | 检查人 | 不通过处理 |
|--------|----------|--------|------------|
| 基础设定定稿 | 10个设定文件是否完整、准确 | 编辑+总编 | 打回修改 |
| 每5章写作 | 是否按章纲、设定写作 | 编辑 | 打回重写 |
| 每5章确认 | 方向是否正确 | 总编 | 暂停纠偏 |
| 每20章反馈 | 整体是否走偏 | 研究员 | 调整后续 |
| 阶段转换 | 本阶段目标是否达成 | 总编+用户 | 暂停/调整 |

## 二、写手自查机制

### 2.1 写作前检查（必须）

每次写作前，写手必须完成以下检查：

```markdown
## 写作前检查清单

### 一、设定文件检查
- [ ] 阅读 outline.md，确认本批次章节在主线中的位置
- [ ] 阅读 world.md，确认本批次涉及的世界观要素
- [ ] 阅读 characters.md，确认出场人物的人设
- [ ] 阅读 skills.md，确认本批次涉及的技能
- [ ] 阅读 level.md，确认实力等级
- [ ] 阅读 factions.md，确认涉及的势力

### 二、章节规划检查
- [ ] 阅读 chapter-plan.md，确认本批次5章的章纲
- [ ] 确认每章的爽点设计
- [ ] 确认每章的钩子设计
- [ ] 确认本批次的伏笔埋设

### 三、长记忆检查
- [ ] 阅读 chapter-summary.md，了解已完成的章节摘要
- [ ] 阅读最近3章正文，确保衔接自然
- [ ] 确认没有偏离已确定的人物关系

### 四、本批次目标确认
- [ ] 本批次要达成什么目标？
- [ ] 这个目标在主线中处于什么位置？
- [ ] 如何确保不偏离目标？
```

### 2.2 写作中检查

每写完一章，立即自查：

```markdown
## 章节自查清单

### 一、基础检查
- [ ] 主角名字是否正确？
- [ ] 配角名字是否正确？
- [ ] 主角性格是否符合 characters.md 定义？
- [ ] 实力等级是否符合 level.md 定义？
- [ ] 技能名称/威力是否符合 skills.md 定义？

### 二、剧情检查
- [ ] 是否按照章纲在写？（如有偏离，立即纠正）
- [ ] 爽点是否到位？
- [ ] 钩子是否埋好？
- [ ] 与前文是否衔接？

### 三、伏笔检查
- [ ] 本章是否需要埋设伏笔？
- [ ] 伏笔是否符合 foreshadowing.md 规划？
- [ ] 之前埋的伏笔是否需要呼应？

### 四、字数/节奏检查
- [ ] 字数是否在 2500-3000 范围？
- [ ] 节奏是否明快？
- [ ] 是否有水字数的地方？
```

### 2.3 写作后检查

每批次（5章）完成后，提交审核前：

```markdown
## 批次提交前检查

### 一、整体检查
- [ ] 5章是否全部完成？
- [ ] 每章字数是否达标？
- [ ] 每章是否有爽点？
- [ ] 每章是否有钩子？

### 二、一致性检查
- [ ] 人物性格是否一致？
- [ ] 战力是否崩坏？
- [ ] 时间线是否正确？
- [ ] 势力关系是否正确？

### 三、伏笔检查
- [ ] 计划埋设的伏笔是否都埋了？
- [ ] 埋设位置是否合理？
- [ ] 是否有意外埋下的伏笔需要记录？

### 四、偏离检查
- [ ] 是否有偏离大纲的地方？
- [ ] 是否有自由发挥的内容？
- [ ] 如有偏离，说明理由
```

## 三、编辑审核机制

### 3.1 审核检查清单

编辑审核时，必须逐项核对：

```markdown
## 编辑审核清单

### 一、一致性核对（必须逐项勾选）

#### 人物一致性
| 检查项 | 第1章 | 第2章 | 第3章 | 第4章 | 第5章 |
|--------|-------|-------|-------|-------|-------|
| 主角名 | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ |
| 主角性格 | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ |
| 配角名 | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ |
| 配角性格 | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ |
| 人物关系 | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ | ✓/✗ |

#### 世界观一致性
| 检查项 | 状态 |
|--------|------|
| 等级体系是否正确 | ✓/✗ |
| 势力分布是否正确 | ✓/✗ |
| 地理设定是否正确 | ✓/✗ |
| 规则设定是否正确 | ✓/✗ |

#### 技能一致性
| 检查项 | 状态 |
|--------|------|
| 技能名称是否正确 | ✓/✗ |
| 技能威力是否一致 | ✓/✗ |
| 进化路线是否正确 | ✓/✗ |

#### 大纲一致性
| 检查项 | 状态 |
|--------|------|
| 剧情是否按大纲发展 | ✓/✗ |
| 爽点是否按计划出现 | ✓/✗ |
| 伏笔是否按计划埋设 | ✓/✗ |
| 章纲是否被严格执行 | ✓/✗ |

### 二、质量检查

| 检查项 | 标准 | 状态 |
|--------|------|------|
| 每章爽点 | 至少1个 | ✓/✗ |
| 每章钩子 | 结尾有悬念 | ✓/✗ |
| 节奏 | 无拖沓 | ✓/✗ |
| 战力 | 不崩坏 | ✓/✗ |
| 字数 | 2500-3000 | ✓/✗ |

### 三、偏离判定

**如有偏离，必须记录**：

| 偏离位置 | 偏离内容 | 偏离原因 | 严重程度 |
|----------|----------|----------|----------|
| 第X章 | 偏离内容 | 原因 | 严重/一般 |

### 四、审核结论

- [ ] **通过**：所有检查项通过
- [ ] **不通过**：存在需要修改的问题

**不通过原因**：
1. xxx
2. xxx
```

### 3.2 打回标准

以下情况必须打回：

| 问题类型 | 严重程度 | 是否打回 |
|----------|----------|----------|
| 人物名字错误 | 致命 | 立即打回 |
| 人物性格与设定不符 | 严重 | 打回 |
| 世界观设定错误 | 严重 | 打回 |
| 技能设定错误 | 严重 | 打回 |
| 战力崩坏 | 严重 | 打回 |
| 偏离大纲主线 | 严重 | 打回 |
| 未按章纲写作 | 一般 | 视情况打回 |
| 字数不足 | 一般 | 打回 |
| 爽点不足 | 一般 | 建议修改 |

## 四、总编监督机制

### 4.1 双重确认

每批次审核通过后，总编必须做第二重确认：

```markdown
## 总编双重确认

### 一、方向检查
- [ ] 本批次是否在主线正确位置上？
- [ ] 与前一批次衔接是否自然？
- [ ] 为下一批次铺垫是否做好？

### 二、大局检查
- [ ] 是否有可能写飞的迹象？
- [ ] 与用户确认的方向是否一致？
- [ ] 节奏是否合理？

### 三、确认结论
- [ ] 确认通过 → 进入下一批次
- [ ] 需要讨论 → 发起全员讨论
- [ ] 需要调整 → 暂停并与用户确认
```

### 4.2 偏离预警

总编需要关注的偏离预警信号：

| 信号 | 含义 | 处理方式 |
|------|------|----------|
| 写手频繁询问"能不能这样写" | 可能有偏离想法 | 及时沟通确认 |
| 某章内容明显加长 | 可能有水字数 | 提醒注意 |
| 爽点明显减少 | 节奏可能放缓 | 提醒加强 |
| 战力出现模糊 | 可能战力崩坏 | 提醒对照 combat.md |
| 突然出现新人物 | 可能脱离设定 | 询问来源 |

### 4.3 纠偏机制

如果发现偏离：

```
发现偏离
    ↓
判断偏离程度
    ↓
轻度偏离 → 编辑审核时指出 → 写手修改
    ↓
中度偏离 → 总编沟通 → 暂停写作 → 讨论确认
    ↓
重度偏离 → 暂停所有工作 → 全员讨论 → 必要时回退
```

## 五、读者反馈机制（每20章）

### 5.1 触发条件

每完成20章，总编**必须**触发读者反馈流程。

### 5.2 反馈内容

研究员作为"读者代表"阅读已完成章节，产出报告：

```markdown
## 读者反馈报告

### 一、整体评估
- 整体评分：X/10
- 是否有走偏：XXX
- 最大的问题：XXX

### 二、大纲遵循检查
- [ ] 主线是否清晰？
- [ ] 支线是否服务于主线？
- [ ] 人物是否崩人设？
- [ ] 战力是否崩坏？
- [ ] 伏笔是否按计划埋设/回收？

### 三、具体问题
| 问题 | 章节 | 描述 |
|------|------|------|
| 问题1 | 第X章 | 描述 |
| 问题2 | 第X章 | 描述 |

### 四、改进建议
1. xxx
2. xxx
```

### 5.3 反馈处理

总编收到反馈后：

1. 分析反馈，识别问题
2. 制定优化方案
3. 将优化意见给到编辑
4. 编辑调整后续审核标准

## 六、长记忆机制

### 6.1 章节摘要

写手必须维护 `chapter-summary.md`，每章完成后更新：

```markdown
## chapter-summary.md

### 第X章
- **核心事件**：XXX
- **出场人物**：XXX
- **实力变化**：XXX级→XXX级
- **爽点**：XXX
- **钩子**：XXX
- **伏笔埋设**：XXX
- **伏笔回收**：XXX（如有）
- **与前文衔接**：XXX

### 第X+1章
...
```

### 6.2 每10章回顾

每10章完成时，写手必须：

1. 阅读 chapter-summary.md 全部内容
2. 确认前后一致性
3. 识别可能的问题
4. 向编辑/总编汇报

## 七、应急预案

### 7.1 轻度偏离

- 编辑审核时指出
- 写手在下一批次中修正
- 记录在案

### 7.2 中度偏离

- 总编介入沟通
- 暂停写作
- 全员讨论确认方向
- 必要时调整大纲

### 7.3 重度偏离（写飞）

- 立即暂停所有工作
- 全员讨论
- 识别从哪个检查点开始走偏
- 决定：回退重写 / 调整大纲 / 腰斩

---

## 八、总结

### 保障体系一览

| 环节 | 保障措施 |
|------|----------|
| **写作前** | 设定文件检查 + 章纲确认 + 长记忆回顾 |
| **写作中** | 每章自查 + 实时对照设定 |
| **写作后** | 批次提交前检查 + 编辑逐项审核 |
| **审核后** | 总编双重确认 + 方向把控 |
| **每20章** | 读者反馈 + 大局纠偏 |
| **全程** | 长记忆维护 + 章节摘要记录 |

**核心原则**：宁可慢，不能偏。一旦发现偏离苗头，立即处理。
